*   **UI Frontend Render:** เมื่อข้อมูลชุดนี้ถูกบันทึกสำเร็จ ข้อมูลใน `interest_tags` จะวิ่งไปถูกแปลงร่างแสดงผลเป็น **"ป้ายแท็กความสนใจ (Interests Tags)"** บนบล็อกโมดูลหน้า Profile (`Module 16.4`) ของยูสเซอร์คนนั้นโดยอัตโนมัติ เพื่อขับเคลื่อนระบบคอมมูนิตี้ให้สแกนหาตัวตนกันได้ง่ายขึ้น[cite: 5]

### 2.1 โครงสร้างและการทำงานของโมดอลต้อนรับ (Modal Flow ด้วย Bootstrap Modal)
เมื่อยูสเซอร์สร้างบัญชีเสร็จแล้ววิ่งเข้ามาที่หน้าโปรไฟล์ครั้งแรก ระบบ React State จะเช็กค่าจากฐานข้อมูลว่ามีค่า `is_first_time: true` หรือไม่ ถ้าใช่ จะทำการสั่งเปิด Bootstrap Modal เด้งขึ้นมาโดยมีเอฟเฟกต์ฉากหลังมืดสลัว (`.modal-backdrop`) และไล่ระดับตามสเต็ปดังนี้:

    * ข้อความหลัก: `“Welcome! Hope you enjoy.”`
    * ปุ่มหลัก: `[Next]` (ใช้คลาส `.btn .btn-primary`) เพื่อกดอัปเดตสเต็กภายใน React Component เข้าสู่หน้าถัดไป
* **โมดอลสเต็ปที่ 2: Add to Home Screen (`Profile-Starter-3.png`)**

    * *เคสผู้ใช้ iOS:* ใช้สถาปัตยกรรม Grid ของ Bootstrap แบ่งพื้นที่สี่เหลี่ยมตรงกลางแสดงภาพกราฟิกหรือ GIF สอนสเต็ปสั้น ๆ (1. กดปุ่มแชร์ด้านล่าง 2. เลือก Add to Home Screen)

## 👤 MODULE 3: PROFILE DASHBOARD & THE DUAL-TAB ARCHITECTURE

### 3.1 โครงสร้างหน้าโปรไฟล์หลัก & Data Binding ด้วย Bootstrap คาร์ด (`Profile-Starter-1.png`, `Profile-2.png`, `Profile-1.png`)
ข้อมูลทั้งหมดในส่วนนี้จะใช้ตรรกะแบบ **"Direct Data Binding"** ผ่าน React Hooks (`useEffect`) ดึงข้อมูลสดจากขั้นตอนสร้างบัญชีและ X Auth มากระจายลง Component โดยไม่ต้องให้ยูสเซอร์พิมพ์ใหม่:
* **Header Section:** พื้นหลัง Cover สี่เหลี่ยมมนขนาดใหญ่ (ใช้คลาส `.img-fluid .rounded`) และวงกลมรูปโปรไฟล์ซ้อนตรงกลางขอบล่าง (ใช้คลาส `.rounded-circle .border .border-3`)
* **Profile Info:**
    * *Name:* แสดงผลชื่อ (Display Name) ตรงตามที่ยูสเซอร์ระบุไว้ตอนสร้างบัญชี
    * *Bio/Description:* ดึงข้อความแนะนำตัวมาจากกล่อง Description มาแสดงผลทั้งหมด (หากไม่ได้กรอกไว้ พื้นที่ส่วนนี้จะซ่อนตัวด้วยเงื่อนไข `{bio && <p>{bio}</p>}`)
    * *ปุ่มสลับมุมมอง (View Action Button):*
        * **Owner View:** หากตรวจสอบแล้ว ID ตรงกับผู้ใช้งานปัจจุบัน จะเรนเดอร์ปุ่ม `[📝 Edit Profile]` (ใช้คลาส `.btn .btn-outline-primary`) สำหรับแก้ไขข้อมูล
        * **Visitor View:** หากเป็นบัญชีอื่นกดเข้ามาดูผ่านหน้าฟีดหรือหน้าแชท ปุ่มแก้ไขนี้จะไม่ถูกเรนเดอร์ลงใน DOM Tree อย่างเด็ดขาด
* **Interests Tags:** ดึงข้อมูลอาร์เรย์ (Array) จากฟิลด์ข้อมูลมาวนลูปด้วย `.map()` ใน React เพื่อเรนเดอร์เป็นป้ายแท็ก (Badges) ด้วยคลาส Bootstrap `.badge .bg-light .text-dark .me-1` แสดงผลใต้ Bio ทันที เช่น `K-Pop`, `T-Pop`, `Thai Actress`
* **Floating Action Button (FAB):** ใช้ปุ่มกดติดสถานะ CSS Fixed `.position-fixed` ของ Bootstrap ลอยอยู่มุมขวาล่างสุดของจอเสมอ (`.bottom-0 .end-0 .m-3`) ในโหมดดูโปรไฟล์ตัวเอง เพื่อให้ยูสเซอร์กดเปิดฟอร์มสร้างโพสต์แจกของใหม่ได้สะดวกรวดเร็ว

### 3.2 Content Navigation (The Dual-Tab Architecture — `Profile-Edit.png`)
ใช้ระบบควบคุมแท็บของ Bootstrap (`.nav .nav-tabs`) ควบคุมสถานะ Active ด้วย React State (`const [activeTab, setActiveTab] = useState('feed')`):
* **แท็บ [Feed]:** สำหรับดูโพสต์/เธรดกิจกรรมแจกของที่ยูสเซอร์คนนี้เคยสร้างขึ้นมาเองทั้งหมด โดยจัดเรียงลำดับตามเวลาล่าสุดจากบนลงล่างอย่างเข้มงวด (**Strict Chronological Query**)
* **แท็บ [Activity]:** สำหรับดูประวัติการทำกิจกรรมย้อนหลังของผู้ใช้ โดยแบ่งออกเป็น 4 หมวดหมู่ย่อยภายในคอมโพเนนต์:
    * *Favorites (Fav — `Profile-3.png`):* รวบรวมโพสต์กิจกรรมที่ผู้ใช้กดถูกใจหรือติดดาวไว้ **[🔒 Visitor Privacy Lock Rule]:** หมวดหมู่นี้จะแสดงให้เห็นเฉพาะเจ้าของโปรไฟล์ (Owner) เท่านั้น หากเป็นบัญชีอื่นกดเข้ามาส่อง (Visitor) เงื่อนไขใน React จะทำการสกัดกั้นและไม่เรนเดอร์แถว Fav นี้ออกไปทันทีเพื่อความเป็นส่วนตัว
    * *Events Participate (`Profile-4.png`):* ประวัติรายชื่ออีเวนต์ที่บัญชีนี้เคยไปกด Check-In ยืนยันตัวตน ณ พิกัดสถานที่จัดงานจริง
    * *Giveaways Backup (`Profile-5.png`):* คลังประวัติโพสต์กิจกรรมแจกของทั้งหมดที่ผู้ใช้รายนี้เคยเปิดแจก (Giver History)
    * *Giveaways Collection (`Profile-6.png`):* บันทึกประวัติของแจกที่บัญชีนี้เคยไปสแกนรับ QR Code หรือที่เคยกดยื่นคำขอนัดแลกเปลี่ยนของกันล่วงหน้าสำเร็จ

## 🧭 MODULE 4: DISCOVER SCREEN & SMART FEED LOGIC

### 4.1 โครงสร้างหน้าจอและองค์ประกอบ UI ด้วย Bootstrap Layout (`Discover-1.png`)
หน้าจอถูกออกแบบด้วยระบบคอนเทนเนอร์และสไตล์การ์ดที่โค้งมนตามหน่วย Base Unit 4px ในสารบบดีไซน์ ประกอบด้วย:

### 4.2 ตรรกะการแสดงผลและกรองข้อมูล (Smart Cold-Start & Date Filtering ใน React)
* **สถานะที่ 1: ยังไม่มีการติดตาม (Zero Marked State):** ถ้าระบบตรวจสอบว่าบัญชีนี้ยังไม่เคยเลือกรีเควสหรือกดติดตาม (Marked) ศิลปินหรืออีเวนท์ใด ๆ เลย ระบบจะดึงข้อมูลศิลปินและอีเวนท์ทั้งหมดในฐานข้อมูลที่มีแท็กตรงกันหรือใกล้เคียงที่สุดกับคีย์เวิร์ดความสนใจที่ยูสเซอร์เลือกไว้ตอน Onboarding มาแสดงผลให้ทั้งหมดเป็นกลุ่มแรก (Suggested Content) เพื่อให้หน้าเว็บดูมีคอนเทนต์พร้อมใช้งานทีนที
* **สถานะที่ 2: เริ่มมีการกดติดตามแล้ว (Marked State):** ทันทีที่ยูสเซอร์กด Mark หรือเลือกติดตามศิลปิน/อีเวนท์ใดอีเวนท์หนึ่งในแอปแล้ว ระบบ React จะสับสวิตช์โฟลวหลังบ้านทันที โดยจะเปลี่ยนมาแสดงผลเฉพาะสิ่งที่คุณกด Marked ไว้เป็นหลัก นำมาจัดเรียงไว้บนสุด ส่วนข้อมูลที่สอดคล้องอื่น ๆ ที่ระบบแนะนำ (Suggested) จะถูกลดลำดับความสำคัญลงไปอยู่ด้านล่างแทน
* **Default Date State:** เมื่อเปิดหน้านี้ขึ้นมา ระบบ React จะทำการเลือกไฮไลท์ (Active) ที่ **"วันที่ปัจจุบัน"** ให้โดยอัตโนมัติ และฟีดการ์ดอีเวนท์ 2 คอลัมน์ด้านล่างจะดึงเฉพาะงานแจกของที่มีพิกัดวันตรงกับวันปัจจุบันมาโชว์ก่อนเป็นอันดับแรก
* **On-Click Filter:** เมื่อยูสเซอร์กดจิ้มเปลี่ยนวันที่บนแถบปฏิทิน (Calendar Strip) ➡️ ระบบ React State จะเปลี่ยนค่าวันที่ ➡️ ฟังก์ชัน `useEffect` จะทำการเรียก Fetch ข้อมูลจาก Supabase ที่มีสถานะวันตรงกับวันที่เลือกใหม่มาเปลี่ยนแสดงผลบนหน้าจอทันทีในเสี้ยววินาที

## 🔍 MODULE 7: CONTEXT-AWARE SEARCH LOGIC

### 7.1 ตรรกะการแบ่งขอบเขตการค้นหา (Search Routing Matrix ใน React)
ระบบค้นหาจะทำงานแตกต่างกันโดยอิงตามตำแหน่งหน้าจอ (Route ของ React Router) ที่ผู้ใช้งานกำลังเปิดอยู่ ดังนี้:
* **เมื่อเสิร์ชจากหน้า Discover (Global Search):** ค้นหาแบบครอบคลุมทั้งหมด วิ่งไป Query ในฐานข้อมูล Supabase พร้อมกันทั้งตาราง `artists` และ `events`
* **เมื่อเสิร์ชจากหน้ารวม Artist (Artist-Only Search) — `Discover-2.png`:** จำกัดขอบเขตให้อยู่ในตาราง `artists` เท่านั้น ผลลัพธ์จะแสดงผลเฉพาะรายชื่อศิลปิน
* **เมื่อเสิร์ชจากหน้ารวม Event (Event-Only Search) — `Discover-3.png`:** จำกัดขอบเขตให้อยู่ในตาราง `events` เท่านั้น ผลลัพธ์จะแสดงผลเฉพาะการ์ดรายการอีเวนท์

### 7.2 โครงสร้างหน้าจอผลลัพธ์และการรับมือเมื่อหาไม่เจอ (No Results Found & UGC Hook ผ่าน Bootstrap UI)
* **A. หน้าค้นหาศิลปิน (Artist Search View — `Discover-2.png`):** แสดงลิสต์รายชื่อศิลปินในรูปแบบแถวแนวตั้ง **Fallback Logic:** หากป้อนตัวอักษรแล้วความยาวอาเรย์ผลลัพธ์เป็น 0 จะเรนเดอร์ข้อความ `“If not found, you can create new.”` คู่กับปุ่ม `[+ New Artist]` (คลาส `.btn .btn-primary`) เพื่อส่งคำขอเพิ่มศิลปินใหม่ได้ทันที

### 9.3 ตรรกะการโต้ตอบและฟีเจอร์ทางลัดบนการ์ดฟีด (Feed Interactions & Quick Actions Logic)
ตรรกะการโต้ตอบ (Interactions) บนการ์ดฟีดแต่ละใบในหน้า Home และหน้า Event Feed มีการทำงานเพิ่มเติม 3 ส่วนหลัก ดังนี้:
1.  **การเปลี่ยนเส้นทางไปยังโปรไฟล์ผู้โพสต์ (Avatar Profile Routing):** เมื่อผู้ใช้กดคลิกที่ รูปโปรไฟล์วงกลม หรือ ชื่อบัญชี (`Account · @account`) ของเจ้าของโพสต์แจกของบนการ์ดฟีด ระบบ React Router จะทำการเปลี่ยนเส้นทางหน้าจอ พายูสเซอร์พุ่งตรงไปยังหน้า User Profile View ของบัญชีนั้น ๆ ทันที
2.  **ปุ่มกดแลกเปลี่ยนด่วนจากหน้าฟีด (Quick Exchange Trigger):** ป้ายแท็กสถานะมุมบนขวาของการ์ดฟีด (เช่น แท็ก Exchange) จะทำหน้าที่เป็นปุ่มกดในตัว ผู้ใช้สามารถกดที่ปุ่ม `[Exchange]` ได้ทันทีตั้งแต่เปิดเจอที่หน้าฟีด โดยระบบ React จะเปิดหน้าต่างโฟลวขอยื่นแลกเปลี่ยนของรางวัล (Exchange Request Flow) ขึ้นมาทันที
3.  **ระบบกดดูรูปภาพเต็มจอ (Image Lightbox Viewer):** เมื่อผู้ใช้กดคลิก (Tap) ที่ รูปภาพหลักของโพสต์แจกของ ระบบ React จะทำงานร่วมกับคอมโพเนนต์เพื่อเปิดโหมด Fullscreen Lightbox Viewer ดึงรูปภาพนั้นขึ้นมาแสดงผลเต็มหน้าจอโทรศัพท์ แผ่นหลังจอจะถูกถมด้วยสีดำสนิทด้วย CSS สไตล์จำเพาะ รองรับการทำ Pinch-to-Zoom และสามารถกดปุ่มปิดป๊อปอัปหรือปัดหน้าจอลง (Swipe Down) เพื่อกลับมาสู่หน้าฟีดเดิมได้อย่างราบรื่น

### 10.1 ฟิลด์ข้อมูลหลักและการกรองอีเวนท์ที่ฉลาด (Event Filter Logic ใน React — `Home-New Post-1.png`)

ฟิลด์ฟอร์มพื้นฐานประกอบด้วย Title/Caption (ใช้แท็ก `<textarea className="form-control">`), Quantity (ea), ปุ่มสลับคลาสแบบกลุ่มของ Bootstrap `[Exchange Switch]`, ตัวระบุ วันที่/เวลา/สถานที่ และแผงเครื่องมือด้านล่างสุด (Bottom Action Bar) สำหรับเลือกแนบรูปภาพ สูงสุด 4 รูป, วิดีโอสั้น, และแนบ URL ลิงก์ภายนอก (`Home-New Post-4.png`, `Home-New Post-6.png`)

### 10.3 ตรรกะการสร้างเธรดเนื้อหาต่อเนื่อง (Thread Creation Feature ด้วย React Component — `Home-New Post-8.png`, `Home-New Post-9.png`)
ใช้ระบบอาเรย์ของ React State บันทึกข้อมูลบล็อกความเห็น เมื่อผู้ใช้กดปุ่มเครื่องหมายบวก `[+]` ฟอร์มก้อนแรกจะถูกย่อส่วนดันขึ้นด้านบนเป็น Preview Card และแตกแขนงแถวเส้นเชื่อมโยงทางซ้าย (ใช้สไตล์ CSS เส้น Border เชื่อมต่อดิ่ง) พร้อมไอคอน `↳ Reply` เพื่อเปิดกล่องข้อความ Caption ชุดที่ 2 (Thread Textarea) สำหรับพิมพ์เธรดต่อพ่วงและแนบสื่อแยกอิสระได้ทันที โดยเมื่อส่งข้อมูลจะยิงบันทึกพร้อมค่า `parent_id` และ `thread_order` ผูกกันในฐานข้อมูลคราวเดียว

### 10.4 ตรรกะการควบคุมสต็อกและการจัดการโควตาแลกเปลี่ยน (Inventory Control & Exchange Reservation Logic)
* **A. นิยามของฟังก์ชัน:** สภาพตั้งต้นเป็นแจกฟรี 100% เสมอ เมื่อเปิดปุ่มเปิดสถานะ `[Exchange]` จะเป็นการเปิดเงื่อนไขรับอาเรย์คำขอให้บัญชีอื่นยื่นคำขอแลกเปลี่ยนเข้ามาพิจารณาได้ โดยไม่กระทบจำนวนสต็อกแจกฟรีภาพรวมในตอนแรก
* **B. ตรรกะการส่งคำขอและการจัดการจำนวน (คำขอแลกเปลี่ยนแบบเสรี):** บัญชีผู้ใช้อื่นสามารถส่งคำขอ (Exchange Request) เข้ามาในระบบได้เรื่อย ๆ อย่างไม่มีการจำกัดจำนวน และจะยังไม่มีการตัดยอดสต็อกหรือกักกันสิทธิ์ใด ๆ ทั้งสิ้นในฝั่งหลังบ้าน จนกว่าตัวเจ้าของโพสต์ (Owner) จะพิจารณาและกดอนุมัติ (Approve) คำขอนั้น ๆ ด้วยตนเองอย่างเป็นทางการผ่านหน้ารวมคำขอแลกเปลี่ยน (`Notification-3.png`)

## 📌 MODULE 11: POST DETAIL DYNAMICS & STATUS SYNCHRONIZATION MATRIX

### 11.1 หน้ารายละเอียดโพสต์และกฎความคงที่ของข้อมูลหลัก (Immutable Post Data Rule — `Home-Post.png`)

* **The Only Exception (Status Badge):** สิ่งเดียวบนหัวโพสต์ที่มีการอัปเดตแบบ Dynamic คือ **ป้ายสถานะ (Status Badge)** ที่ใช้คลาสสลับสีของ Bootstrap (เช่น `.badge .bg-warning` เป็น `.bg-success`) ซึ่งจะซิงค์เปลี่ยนตามการกด Reply ประเภท Ready (พร้อมแจก) หรือ End (ของหมด) ของเจ้าของโพสต์เท่านั้น

### 11.2 ระบบประเภทการตอบกลับสามสถานะ (The 3-Type Reply System — `Home-Post-2.png`)
บริเวณกล่องข้อความตอบกลับด้านล่างสุด จะมีปุ่มดรอปดาวน์เลือกประเภทการ Reply 3 รูปแบบ:
1.  **Reply (ธรรมดา) [`Home-Post Reply-1.png`]:** พูดคุย สอบถาม ทั่วไป ไม่มีผลต่อสถานะหรือหมุดแผนที่

3.  **End (ของหมดแล้ว) [`Home-Post Reply-3.png`]:** พิมพ์ปิดโพสต์ ➡️ สั่งอัปเดตสถานะหัวโพสต์ผ่าน React และฐานข้อมูลเป็น [End] และลบ/ซ่อนหมุดโพสต์นี้ออกจากหน้าแผนที่ทันที

## 🧭 MODULE 13: INBOX & PRIVATE CHAT SYSTEM SPECS

โมดูลระบบกล่องข้อความแชทส่วนตัว (Private Message) ที่ออกแบบมาเพื่อรองรับการสื่อสาร แลกเปลี่ยนข้อมูล และการนัดหมายส่งมอบของแจกหน้างานคอนเสิร์ต/อีเวนต์ระหว่างผู้ใช้งานในระบบอย่างมีประสิทธิภาพ

### 13.1 หน้ารวมแชท (Inbox Chat List — `Inbox-1.png`)

#### 🍀 Clover Badge Icon Assignment (สัญลักษณ์การแลกเปลี่ยน):
* **Logic:** ในรายการห้องแชทรวม หากแชทของคู่สนทนาคนใดมีจุดเริ่มต้นหรือมีความเกี่ยวพันมาจาก "การทำรายการคำขอแลกของแจก (Exchange Transaction)" ระบบจะต้องทำการแสดงผล ไอคอนรูปใบโคลเวอร์ (Clover Icon) ขึ้นมากำกับเป็น Badge พิเศษ
* **UI Implementation Note:** ตามคำแนะนำของคุณดีไซเนอร์ ในขั้นตอนการพัฒนา UI จริง ให้เปลี่ยนตำแหน่งการแสดงผลไอคอนใบโคลเวอร์นี้ไปวางเป็น **Badge ซ้อนทับอยู่บนมุมรูปโปรไฟล์ (Avatar Profile Badge)** ของผู้ใช้รายนั้น ๆ แทนไอคอนขวาล่าง เพื่อความสวยงามและประหยัดพื้นที่หน้าจอ
* **Zero-State Rule:** ห้องแชททั่วไปที่ไม่ได้มาจากการแลกของแจก (คุยทั่วไปหรือรับของปกติที่ไม่มีการยื่นหมูยื่นแมว) จะไม่มีสัญลักษณ์ใบโคลเวอร์นี้ปรากฏบนรูปโปรไฟล์โดยเด็ดขาด

### 13.2 ส่วนแสดงข้อมูลของแจกภายในห้องแชท (Collapsible Transaction Header)

เมื่อกดเข้าไปในห้องแชทที่มีการทำรายการแลกของกัน ระบบจะจัดการแสดงผลส่วนหัว (Header) เพื่อเตือนความจำให้ทั้งสองฝ่ายทราบว่ากำลังทำธุรกรรมอะไรกันอยู่ โดยมีสถานะการแสดงผล 2 รูปแบบดังนี้:

#### 1. Expanded State (สถานะเปิดแสดง — `Chat-1.png`):
* ระบบจะดึงข้อมูลการ์ดสิ่งของแจกของอีกฝ่ายขึ้นมาโชว์ที่ด้านบนสุดของหน้าต่างแชท เพื่อระบุชัดเจนว่า *"เรากำลังแลกเปลี่ยนอะไรกับเขา"* พร้อมแสดงปุ่มสถานะกำกับ (เช่น `[Receive]`)
* มีปุ่มลูกศรชี้ลง `∨` สำหรับสั่งพับเก็บหน้าต่าง

#### 2. Collapsed State (สถานะพับเก็บ — `Chat-2.png`):
* เมื่อกดปุ่มลูกศร แถบข้อมูลการ์ดด้านบนจะถูกยุบพับเก็บเหลือเพียงแถบ Bar สั้น ๆ เขียนว่า *"Giveaway"* พร้อมไอคอนลูกศรเปลี่ยนเป็นชี้ขึ้น `∧` เพื่อประหยัดพื้นที่บนหน้าจอในการพิมพ์ข้อความสนทนา

#### 🚫 Zero-State Rule:
ในกรณีที่สองฝ่ายเปิดแชทคุยกันโดยที่ไม่มีความเกี่ยวข้องกับการแลกเปลี่ยนของแจกใด ๆ เลย (General Chat) ระบบจะต้องทำการ**ซ่อน (Hide) แถบประวัติการ์ดส่วนหัวนี้ออกไปโดยสิ้นเชิง** ไม่ให้ปรากฏขึ้นมารบกวนพื้นที่หน้าจอ

### 13.3 แถบเครื่องมือแนบสื่อและการปักหมุดนัดหมาย (Media Attachment Sheet & Pin Location)

ที่บริเวณกล่องพิมพ์ข้อความ (Chat Input Box) เมื่อผู้ใช้กดปุ่มเครื่องหมายบวก `+` ระบบจะทำการเลื่อนแถบ Action Menu Tab ขึ้นมาจากด้านล่างเพื่อเปิดฟังก์ชันการส่งข้อมูลสื่อสารแบบครบวงจร (`Chat-2.png`) รองรับสถานการณ์นัดหมายด่วนหน้างาน:

* **เมนูที่ 1: คลังรูปภาพ (Gallery — `Chat-3.png`)**
    * *Use Case:* สำหรับใช้ถ่ายรูปหน้างานหรืออัปโหลดรูปภาพส่งให้คู่สนทนาดูว่าตอนนี้เรายืนอยู่ตรงไหนของฮอลล์ หรือแต่งตัวสีอะไรอยู่เพื่อให้หากันเจอได้ง่ายขึ้น
* **เมนูที่ 2: วิดีโอ (Video — `Chat-3.png`)**
    * *Use Case:* สำหรับส่งคลิปวิดีโอบรรยากาศสั้น ๆ รอบตัวในพิกัดนัดหมายเพื่อระบุตำแหน่งจริง
* **เมนูที่ 3: ไฟล์เอกสาร (Document — `Chat-4.png`)**
    * *Use Case:* สำหรับใช้ดาวน์โหลด/อัปโหลดไฟล์เอกสารยืนยันสิทธิ์หรือไฟล์หลักฐานต่าง ๆ ที่จำเป็นในระบบ

    * **Core Utility:** แผงควบคุมมีช่องอินพุตค้นหาและบันทึกพิกัดสถานที่ (Location) ลงในแชท
    * **Use Case:** ฟีเจอร์นี้ทำมาเพื่อแก้ Pain Point สำคัญ กรณีที่แฟนคลับเข้ามารับของแจกตามพิกัดเดิมไม่ทันในเวลาที่สถานะ Ready บังคับ (เช่น ประตูฮอลล์เปิดก่อน หรือรถติดเดินทางมาเลท) ผู้ใช้ทั้งสองฝ่ายสามารถใช้ฟังก์ชันนี้ในการส่งพิกัด Map Pin เพื่อทำการนัดหมายสถานที่นัดรับของแห่งใหม่ **(New Private Appointment Point)** เช่น นัดเจอกันที่สถานีรถไฟฟ้าใกล้เคียง หรือล็อบบี้โรงแรมข้าง ๆ แทนได้อย่างแม่นยำและเป็นส่วนตัว

### 13.4 ระบบเปิด-ปิดแจ้งเตือนรายบุคคล/รายห้องแชท (Per-Chat Notification Toggle)
* **ปุ่มควบคุมบนหน้าจอ (UI Component):** ภายในหน้าห้องแชท (Chat Room) ที่มุมขวาบนของแถบ Header (ข้าง ๆ ปุ่ม Setting หรือจุดไข่ปลา) ให้เรนเดอร์ **"ปุ่มไอคอนรูปกระดิ่ง" (Bell Icon)** สำหรับควบคุมการแจ้งเตือนเฉพาะห้องแชทนั้น ๆ
* **สถานะและการทำงาน (Toggle States Logic):** ควบคุมไอคอนและการทำงานผ่าน React State (`const [isMuted, setIsMuted] = useState(false)`) ผูกเข้ากับอีเวนต์คำสั่ง `onClick`:
    * **สถานะเปิดแจ้งเตือน (Default — Unmuted):** แสดงไอคอนกระดิ่งปกติ เมื่อมีข้อความใหม่เข้าจากห้องแชทนี้ ระบบจะยิง Push Notification และแสดงจุดแจ้งเตือนตามปกติ
    * **สถานะปิดแจ้งเตือน (Muted):** เมื่อผู้ใช้กดที่ปุ่ม ไอคอนกระดิ่งจะเปลี่ยนคลาสไอคอนเป็น **"กระดิ่งมีเส้นคาดขวับ" (Bell-Off Icon)** พร้อมส่งคำสั่งไปเซ็ตค่าลงคอลัมน์ในตารางห้องแชทรายบุคคล
* **Backend & App Logic:** ทันทีที่ห้องแชทใดถูกสลับเป็นสถานะ Muted ระบบส่วนหลังจะต้องระงับการส่ง Push Notification ของห้องแชทนี้มายังเครื่องผู้ใช้ทันที แต่ตัวเลขสะสมข้อความที่ยังไม่ได้อ่าน (Unread Badge) ในหน้าต่างรวมแชท (Chat List) ที่เป็นป้าย Bootstrap `.badge .bg-danger` จะยังคงนับและแสดงผลตามปกติ เพื่อให้ผู้ใช้รับรู้ว่ามีข้อความเข้าเมื่อเปิดแอปเข้ามาดู

## 📌 MODULE 14: QR VERIFICATION & STOCK INVENTORY SPECS (EXCHANGE MODE)

### 14.1 การเข้าสู่หน้ายืนยันสิทธิ์ (Entry Points & Status Change)
* **Trigger Condition:** ระบบ React จะเช็กและเปิดเงื่อนไขเรนเดอร์ให้เข้าสู่กระบวนการยืนยันสิทธิ์แลกเปลี่ยน ก็ต่อเมื่อสถานะของโพสต์แจกของใน State เปลี่ยนเป็น **"Ready"** (พร้อมแจกหน้างาน) แล้วเท่านั้น
* **Entry Points:** สามารถเข้าถึงหน้าจอนี้ได้จาก 3 ช่องทางหลัก:
    1. การกดที่หมุด (Pin) ของโพสต์นั้น ๆ บนแผนที่ (Map View)
    2. การกดปุ่มบนหน้าโพสต์รายละเอียด (Post Detail) ซึ่งปุ่มเดิมที่เป็นคำว่า `[Exchange]` จะถูกถอดออกและสลับเปลี่ยนสถานะเป็นปุ่ม **`[Receive]`** สำหรับฝั่งผู้ขอแลก และปุ่ม **`[Scan]`** สำหรับฝั่งผู้แจก (ใช้คลาสสีปุ่มตัดกันเด่นชัด)
    3. การกดผ่านลิงก์สถานะธุรกรรมในหน้ารายรูมแชท (Private Chat Room Header)

### 14.2 ฝั่งผู้ขอแลก (Receiver Interface — `Giveaway-Exchange-Receiver-1.png`, `Giveaway-Exchange-Receiver-2.png`)
* **หน้าจอหลักของผู้ขอแลก (`Giveaway-Exchange-Receiver-1.png`):** ระบบจะแสดงการ์ดรายละเอียดการจับคู่แลกเปลี่ยนชัดเจนด้วย Bootstrap Layout ว่าใครให้ชิ้นไหน และใครแลกชิ้นไหนกลับไป พร้อมข้อความกำกับ *“You are receiver. Please show your receive QR to giver when you exchange your giveaways.”* และมีปุ่มไอคอน QR Code ด้านล่างสุด
* **ระบบถอดรหัส QR Code (`Giveaway-Exchange-Receiver-2.png`):**
    * เมื่อผู้ขอแลกกดที่ไอคอน QR ระบบจะทำการเปิด Bootstrap Modal เจเนอเรต **"Receive QR"** ขึ้นมา (ใช้ไลบรารีสากลของ React เช่น `qrcode.react`) ในรูปแบบหน้าต่างป็อปอัป
    * **Permanent QR Logic (สิทธิ์ถาวร):** เนื่องจากเป็นการจับคู่แลกของแบบเจาะจงตัวบุคคลไว้ล่วงหน้า (Exchange Transaction) QR นี้จะไม่มีการกำหนดเวลาหมดอายุ (No Time Limit) ตัวเลขเคาน์เตอร์นับถอยหลังใน React UI จะถูกฟิกซ์เป็นสตริงค่ายอดฮิต `00 : 00` เสมอ โดย QR จะคงอยู่และใช้งานได้ตลอดไปจนกว่าจะถูกฝั่งผู้แจกสแกนใช้งานสำเร็จเรียบร้อยเท่านั้น

### 14.3 ฝั่งผู้แจกแลกเปลี่ยน (Giver Interface & Inventory Counter — `Giveaway-Exchange-Giver-1.png`, `Giveaway-Exchange-Giver-2.png`)
* **หน้าจอหลักของผู้แจก (`Giveaway-Exchange-Giver-1.png`):** จะแสดงโครงสร้างเลย์เอาต์คล้ายกับฝั่งผู้ขอแลก แต่ระบบจะเรนเดอร์ข้อความกำกับเป็น *“You are giver. Please scan receive QR from receiver...”* และเปลี่ยนไอคอนด้านล่างเป็น **ปุ่มสแกน (Scan Icon)** แทน
* **หน้าจอกล้องสแกนและโควตาคงเหลือ (`Giveaway-Exchange-Giver-2.png`):**
    * เมื่อกดปุ่มสแกน ระบบ React จะเปิด Modal กล้อง QR Scanner ขึ้นมา (เรียกใช้โมดูลสแกนกล้อง เช่น `@html5-qrcode` หรือคอมโพเนนต์อื่น ๆ ที่เข้ากันได้)
    * **Reserved Stock Counter Logic (การนับยอดกั๊กเทียบยอดรวม):** ที่ด้านล่างของกรอบสแกน หน้าจอต้องแสดงตัวเลขโควตาสินค้าในรูปแบบ **`[จำนวนที่เหลืออยู่ของที่กั๊กไว้สำหรับแลก] / [จำนวนของแจกทั้งหมด]`** (เช่น `05 / 20`) ด้วยฟอนต์สไตล์ Monospace ตัวหนาเด่นชัด
    * **Business Rule:** ตัวเลขฝั่งซ้าย (Numerator) จะนับถอยหลังเฉพาะโควตาที่ล็อกไว้สำหรับเคสแลกเปลี่ยนเท่านั้น ส่วนตัวเลขฝั่งขวา (Denominator) จะเป็นจำนวนของแจกทั้งหมดในโพสต์ (รวมทั้งแบบแจกฟรีทั่วไปและแบบแลกเปลี่ยน) เพื่อให้ผู้แจกเช็กได้ทันทีว่าโควตาแลกเปลี่ยนที่เหลืออยู่ คิดเป็นสัดส่วนเท่าไหร่จากของทั้งหมดที่มี

### 14.4 Status & Stock Management (Success Handling — `Giveaway-Exchange-Giver-3.png`, `Giveaway-Exchange-Receiver-5.png`, `Giveaway-Exchange-Giver-4.png`)
* **การอัปเดตสถานะ Real-time (`Giveaway-Exchange-Giver-5.png`):** ทันทีที่กล้องฝั่งผู้แจกถอดรหัสและตรวจสอบเงื่อนไขความถูกต้องผ่านโค้ดใน React เรียบร้อยสำเร็จ ระบบจะสลับ Component UI ขึ้นเครื่องหมายติ๊กถูกสีเขียว (ใช้แอนิเมชันหรือคลาสความสำเร็จยอดนิยม) พร้อมข้อความระบุชัดเจนว่า *“Exchange complete!”*
* **Stock Deduction (การตัดยอด) (`Giveaway-Exchange-Giver-2.png`):** ระบบจะทำการตัดยอดตัวเลขอินเวนทอรีฝั่งคงเหลือออกทันที 1 สิทธิ์ (เช่น จาก `05 / 20` สลับค่าใน React State เปลี่ยนเป็น `04 / 20`) ในเสี้ยววินาทีนั้น พร้อมส่งคำสั่งลดจำนวนลงฐานข้อมูลแบบอะซิงโครนัส
* **End-State Feedback (เมื่อแจกครบ):**
    * สำหรับฝั่งผู้รับ (`Giveaway-Exchange-Receiver-3.png`): จะมีข้อความเช็คสถานะผ่านสตรีมข้อมูล และเปิดป็อปอัปแจ้งเตือนลอยขึ้นมาด้วยสไตล์ Toast หรือ Modal ว่า *“You have received giveaway. Congratulations!”*
    * สำหรับฝั่งผู้ให้เมื่อโควตาหมด (`Giveaway-Exchange-Giver-4.png`): หากสแกนเครดิตชิ้นสุดท้ายเสร็จสิ้นและยอดคงเหลือกลายเป็น 00 บล็อกเงื่อนไขใน React จะทำการเปิดป็อปอัปปิดงานแสดงคำชมเชยว่า *“You have gave all giveaway. Congratulations!”* เพื่อจบลูปธุรกรรมอย่างสมบูรณ์

## 📌 MODULE 15: QR VERIFICATION & STOCK INVENTORY SPECS (GENERAL CLAIM MODE)

### 15.1 โครงสร้างหน้าจอและทางเข้า (UI Navigation & Entry Points ผ่าน Bootstrap คาร์ด)
* **ฝั่งคนมารับ (Receiver — `Giveaway-Receiver-1.png`):** หน้าจอจะไม่แสดงการจับคู่สิ่งของ แต่จะวาดโครงสร้างแบบสะอาดตาบอกเพียงข้อมูลสากลว่า "เรามารับของจากโพสต์ไหน" (แสดงข้อมูลการ์ดภาพโปสเตอร์ Event และข้อมูลผู้แจกในเซกชัน Giver) พร้อมข้อความ *“You are receiver. Please show your receive QR to giver when you exchange your giveaways.”* และมีปุ่มไอคอนกลมสำหรับเปิด QR ด้านล่าง
* **ฝั่งคนแจก (Giver — `Giveaway-Giver-1.png`):** หน้านี้จะเป็นหน้าหลัก (Main Hub) ของฝั่งผู้แจกเมื่อกดเข้ามาจากโพสต์ของตัวเอง โดยระบบจะแสดงคำว่า "My Giveaway" ในคาร์ดสินค้า และขึ้นข้อความกำกับชัดเจนว่า *“You are giver. Please scan receive QR from receiver...”* พร้อมปุ่มเปิดกล้องสแกนขนาดใหญ่ด้านล่าง

### 15.2 ลอจิกการควบคุมอายุ QR (Dynamic QR Code Timing Logic ผ่าน React Timer Hooks)
* **หน้าต่างเจเนอเรต QR Code (`Giveaway-Receiver-2.png`):** เมื่อฝั่งคนรับกดปุ่มเปิด QR ระบบจะแสดง Modal "Receive QR" พร้อมเรนเดอร์ตัวนับเวลาถอยหลัง (Countdown Timer โดยใช้ฟังก์ชัน `setInterval` ในคอมโพเนนต์ React Hook)
* **QR Validity (อายุ 2 นาที):** QR สำหรับการรับปกติจะมีอายุการใช้งานเพียง 2 นาที เริ่มต้นที่ตัวเลขค่าจากรัฐสแต็ก `02:00` และนับถอยหลังลงเรื่อย ๆ ต้องทำการสแกนภายในเวลาหน้างานทันที
* **Timeout Cooldown (บล็อก 5 นาที — `Giveaway-Receiver-4.png`):** หากผู้รับสแกนไม่ทันภายใน 2 นาที โทเค็นภายในโครงสร้างจะหมดอายุลง และระบบ React จะสลับเปลี่ยน UI แสดงหน้าแจ้งเตือนความล้มเหลวระบุข้อความ *“Time over. You can request another receive QR in 5 minutes.”*
* **Business Rule:** เพื่อป้องกันไม่ให้ผู้ใช้กดเจนฯ QR ทิ้งไว้ล่วงหน้าเพื่อจองคิว ระบบจะเซ็ตค่าบล็อกบนเครื่องและฐานข้อมูล ไม่ให้ผู้ใช้คนนั้นสามารถกดขอคลิกปุ่มเจเนอเรต Receive QR ตัวใหม่ได้จนกว่าจะครบกำหนดเวลาคูลดาวน์ 5 นาทีเต็ม

### 15.3 ลอจิกการนับสต็อกแบบผสมและจุดสิ้นสุด (Hybrid Live-Inventory Display & Scan Guard)
1. **สูตรการคำนวณยอดแสดงผลบนหน้าจอ (The Live Counter Formula):**
   ใช้ระบบแสดงผลแบบเศษส่วนเพื่อให้เห็นยอดคงเหลือรวมเทียบกับยอดตั้งต้นบริเวณด้านล่างของกรอบสแกนฝั่งผู้แจก (`Giveaway-Giver-2.png`):
   * **ตัวเลขฝั่งซ้าย (Numerator):** $$\text{ยอดคงเหลือจริง} = (\text{จำนวนกั๊กแลกที่เหลืออยู่}) + (\text{จำนวนแจกฟรีที่เหลืออยู่})$$
   * **ตัวเลขฝั่งขวา (Denominator):** $$\text{ยอดรวมทั้งหมด (Total Pool)}$$
   *(ตัวอย่างเช่น แจกฟรีเหลือ 15, นัดแลกเหลือ 5 รวมแสดงฝั่งซ้ายเป็น 20 / 20 ในตอนเริ่มต้น)*

2. **ตัวอย่างลำดับการตัดสต็อก (Scenario Walkthrough):**
   *(สมมติยอดรวม 20 ชิ้น | กั๊กแลก 5 | แจกฟรี 15)*
   * **สถานะเริ่มต้น:** แสดงผลในกล่องข้อความว่า `20 / 20`
   * **จังหวะที่ 1:** แจกฟรีไป 2 คน ➡️ ระบบหักส่วนแจกฟรีออก 2 ในฐานข้อมูลและ React State ➡️ แสดงผลเปลี่ยนเลขเป็น `18 / 20`
   * **จังหวะที่ 2:** มีคนนัดแลกมาใช้สิทธิ์ 1 คน (สแกนพร้อมกันได้ไม่สะดุด) ➡️ ระบบหักส่วนกั๊กออก 1 ➡️ แสดงผลเปลี่ยนเลขเป็น `17 / 20`
   * **จังหวะที่ 3:** เมื่อสแกนแจกฟรีไปจนครบ 15 ชิ้น (โควตาฟรีหมด) และมีการสแกนแลกไปแล้ว 1 คน ยอดคงเหลือจะเหลือเฉพาะของที่กั๊กไว้แลกอีก 4 ชิ้น ➡️ ระบบคำนวณและแสดงผล `04 / 20`

3. **กฎความปลอดภัยและการป้องกัน (System Guardrails ใน React Handler):**
   * **Automatic Rejection:** ทันทีที่โควตา "แจกฟรี" หมดลง (ตัวเลขฝั่งซ้ายคำนวณลดลงมาค้างอยู่ที่ยอดคงเหลือของฝั่งนัดแลกพอดี) ฟังก์ชันตรวจสอบใน React Scanner จะทำการสกัดกั้น คืนค่าบล็อกและปฏิเสธธุรกรรมรหัส QR Code ประเภท 2 นาที (General Claim) ทันที เพื่อเก็บของที่เหลือไว้ให้คนที่นัดแลก (Exchange) เท่านั้น
   * **Validation:** หากมีคนทั่วไปยื่น QR มาสแกนหลังจากนั้น ระบบจะไม่สามารถตรวจสอบผ่านและไม่มีกระบวนการตัดยอดใด ๆ เกิดขึ้น โดยจะระบุข้อความ *“Sorry, This giveaway is out of stock.”* ระบบจะอนุญาตให้สแกนผ่านเฉพาะรหัสโครงสร้างที่เป็นประเภท QR Code ถาวรที่มาจากการนัดหมายแลกเปลี่ยนล่วงหน้าเท่านั้น จนกว่ายอดรวมฝั่งซ้ายจะถูกหักกลบลดลงเหลือเป็นค่าตัวเลข `00` อย่างสมบูรณ์ (`Giveaway-Receiver-6.png`)

### 15.4 ระบบความปลอดภัยและการผูกรหัสโพสต์ (Post-Specific Validation & Safety Rule)
* **Post-Specific Binding Token:** ทุกครั้งที่ฝั่งผู้รับกดขอ "Receive QR" จากหน้าโพสต์แจกของใด ๆ (`Giveaway-Receiver-1.png`, `Giveaway-Receiver-2.png`) ระบบส่วนหลัง (Backend) จะทำการฝังรหัสประจำตัวของโพสต์นั้น (`Post_ID`) และรหัสผู้แจก (`Giver_ID`) เข้าไปใน Token ของ QR Code ตัวนั้นด้วยเสมอ
* **Cross-Post Protection (ป้องกันการสแกนข้ามโพสต์):** เมื่อฝั่งผู้แจกเปิดกล้องสแกนเพื่อแจกของจากโพสต์ของตนเอง ตัวตรวจสอบภายในแอปพลิเคชัน React จะทำการดึงค่า `Post_ID` ที่กำลังเปิดอยู่มาทำการจับคู่ตรวจสอบเงื่อนไขแบบตรงกัน (Exact Match Check) ทันทีกับข้อมูลที่ถอดรหัสได้จากกล้องสแกน
* **กฎระบบ:** หากผู้รับนำ QR Code ที่กดเจนมาจากโพสต์อื่นที่รหัสไม่ตรงกันมายื่นให้ผู้แจกคนนี้สแกน ตัวคอมโพเนนต์จะประมวลผลลัพธ์เป็นความผิดพลาด สั่งแสดงกล่องข้อความเตือนความล้มเหลวข้ามประเภท และจะไม่มีการสั่งยิง API เพื่อตัดยอดสต็อกในระบบข้อมูลเด็ดขาด ของแจกจะไม่สามารถสแกนข้ามโพสต์กันได้เพื่อความปลอดภัยสูงสุด

### 15.5 ตรรกะการตรวจสอบตำแหน่งพิกัดหน้างานและการควบคุมสิทธิ์สำหรับ General Claim Mode (Geofencing GPS Verification Guardrail)

เพื่อป้องกันปัญหาการทุจริต (Anti-Fraud) จากกลุ่มผู้ใช้งานที่ไม่ได้เดินทางมาเข้าร่วมงานอีเวนต์จริง แต่อาศัยการกดรับสิทธิ์หรือเจนเนอเรตคิวอาร์โค้ดจากระยะไกลเพื่อกั๊กของรางวัล (General Claim Mode) ระบบฐานข้อมูลและโมดูลตรวจรับสิทธิ์ส่วนหน้า (Frontend Claim Handler) จะต้องบังคับใช้กฎเหล็กด้านพิกัดภูมิศาสตร์ (Geofencing Rules) ดังต่อไปนี้:

#### 1. ข้อกำหนดและรัศมีระยะห่างที่อนุญาต (Geofencing Validation Rules)
* **The Validation Rule:** ทันทีที่ผู้ใช้งานฝั่งผู้รับ (Receiver) กดปุ่มเพื่อดำเนินการขอรับของแจกฟรี หรือสั่งเจนเนอเรตคิวอาร์โค้ดสัญญาสิทธิ์ ตัวแอปพลิเคชันจะเปิดฟังก์ชันเรียกขอพิกัดปัจจุบันผ่านโปรโตคอล `navigator.geolocation` ของเว็บบราวเซอร์อุปกรณ์มือถือ
* **การคำนวณระยะทาง (Haversine Formula):** ระบบจะนำพิกัดละติจูดและลองจิจูดในปัจจุบันของผู้ใช้ ($GPS_{User}$) วิ่งไปคำนวณเปรียบเทียบระยะทางกับพิกัดสถานที่นัดรับของที่ฝั่งผู้แจกปักหมุดไว้ล่าสุดในหน้ารายละเอียดสิ่งของแจก ($GPS_{Giver}$) 
* **รัศมีปลอดภัยควบคุมระบบ (Allowed Radius):** * ผู้รับจะสามารถดำเนินกระบวนการเจนเนอเรต QR Code สำเร็จก็ต่อเมื่อระยะห่างระหว่างผู้รับและผู้แจกมีค่า **น้อยกว่าหรือเท่ากับรัศมีพิกัดที่กำหนดไว้ในเงื่อนไขสิ่งของแจกนั้น ๆ (เช่น ภายในระยะ 200 เมตร, 500 เมตร หรือตามขนาดพื้นที่ของสถานที่จัดงาน)**
    * สูตรสเปกตัวควบคุมตรรกะเบื้องหลัง:
      $$\text{Distance}(GPS_{User}, GPS_{Giver}) \le \text{Allowed Radius}$$

#### 2. วิธีการตอบสนองของระบบเมื่อตรวจพบว่าอยู่นอกพื้นที่ (Fallback & Error Interface)
หากผลการคำนวณระบุว่าผู้ใช้คนดังกล่าวยู่นอกพื้นที่รัศมีพิกัดหน้างาน ระบบจะต้องดำเนินการล็อกและแสดงผลดังนี้:
* **UI Action:** ระบบจะทำการล็อกสถานะปุ่มกดรับสิทธิ์บนหน้าจอให้กลายเป็นสีจาง (Disabled State) ไม่ยอมให้เกิดการเจาะระบบส่งข้อมูลสร้างคิวอาร์โค้ดขึ้นมาบน DOM
* **Error Alert Box / Notification Window (`Giveaway-Receiver-3.png`):** แสดงกล่องข้อความเตือนความปลอดภัยบนหน้าจอระบุว่า: 
  > *"You are currently too far from the giveaway location. Please approach the designated area."*
* **Exception Rule for Exchange Mode (ข้อยกเว้นสำหรับโหมดแลกเปลี่ยน):** เงื่อนไขการบล็อกด้วยพิกัด Geofencing นี้ จะถูกนำมาบังคับใช้กับระบบ **General Claim Mode (รับฟรี)** เพื่อกันการสแปมสต็อกของเป็นหลักเท่านั้น หากธุรกรรมใดเป็นประเภท **Exchange Mode (โหมดนัดแลกของ)** ที่สองฝ่ายคุยแชทและยินยอมตัดสต็อกแมตช์สิทธิ์กันไปแล้วล่วงหน้า ระบบจะ **ปิดการล็อก Geofencing นี้โดยอัตโนมัติ** เพื่อให้ผู้แลกเปลี่ยนสามารถแสดงสิทธิ์คิวอาร์นัดหมายได้อย่างยืดหยุ่นแม้จะติดปัญหาเรื่องพิกัดหน้างานก็ตาม (โดยให้ไปอ้างอิงตรรกะ Private Appointment พิกัดใหม่ในโมดูลแชท 13.3 แทน)

## 🧭 MODULE 16: BOOKMARKS HUB & CATEGORIZED LISTS NAVIGATION

ระบบศูนย์รวมและจัดการข้อมูลที่ผู้ใช้กดบันทึก (Marked) เก็บไว้ โดยแบ่งหมวดหมู่และการนำทางอย่างเป็นระบบเพื่อความรวดเร็วในการเข้าถึงข้อมูลหน้างานคอนเสิร์ต/อีเวนต์

### 16.1 หน้าแรกสุดของบุ๊กมาร์ก (Bookmarks Main Hub — `Bookmarks-1.png`)
* **Overview:** เมื่อผู้ใช้กดแท็บ Bookmarks บน Bottom Navigation Bar จะพบกับหน้าศูนย์รวมที่แบ่งข้อมูลที่ผู้ใช้เคยทำการ "Marked" (กดเซฟ) ไว้เป็น 3 ส่วนหลักในแนวตั้ง: Artists, Events, และ Giveaways
* **Section 1: Artists (ศิลปิน)**
    * แสดงผลในรูปแบบ Grid แนวนอน (Horizontal List) เป็นวงกลมรูปโปรไฟล์ศิลปิน
    * **Navigation Logic:** หากกดที่ตัวศิลปินโดยตรง หรือกดปุ่มลูกศร `>` ที่หัวข้อสัญลักษณ์ จะนำทางผู้ใช้ไปยัง *"หน้ารวม Artists Bookmark"* (`Bookmarks-Artist-1.png`)
* **Section 2: Events (อีเวนท์)**
    * แสดงผลการ์ดสี่เหลี่ยมจัตุรัสจัดเรียงแบบ Grid (มีปุ่ม `+X ชิ้นที่เหลือ` ที่มุมขวาล่าง)
    * **Navigation Logic:** หากกดที่การ์ดอีเวนท์ใด ๆ หรือกดปุ่มลูกศร `>` จะนำทางผู้ใช้ไปยัง *"หน้ารวม Events Bookmark"* (`Bookmarks-Event-1.jpg` / `Bookmarks-Event-2.png`)
* **Section 3: Giveaways (ของแจก)**
    * แสดงผลการ์ดรูปแบบเดียวกับส่วนของ Events พร้อมปุ่มดูเพิ่มเติม `+X`
    * **Navigation Logic:** หากกดที่การ์ดของแจก หรือกดปุ่มลูกศร `>` จะนำทางผู้ใช้ไปยัง *"หน้ารวม Giveaways Bookmark"* (`Bookmarks-Giveaway-1.png` / `Bookmarks-Giveaway-2.jpg`)

### 16.2 หน้ารวมศิลปินที่บันทึกไว้ (Artists Bookmarks — `Bookmarks-Artist-1.png`)
* **UI Layout:** แสดงผลเป็น Grid การ์ดสี่เหลี่ยมแนวตั้ง 3 คอลัมน์ ด้านบนมีปุ่มตัวเลือกเปิด Dropdown สำหรับจัดเรียงข้อมูล (Sort by Name, Marked Date) และมีปุ่มฟิลเตอร์ประเภทศิลปินด้านข้าง (เช่น `Solo Artist`, `Band`, `Actor`)
* **Bookmark Icon:** บนการ์ดทุกใบจะมีไอคอนริบบิ้นบุ๊กมาร์ก (Bookmark Ribbon) แสดงสถานะกำกับไว้ชัดเจน
* **Deep Navigation:** เมื่อผู้ใช้กดเลือกศิลปินคนใดคนหนึ่งในหน้านี้ ระบบจะนำทางเชื่อมต่อไปยัง *"หน้าโปรไฟล์ศิลปิน (Artist Profile View)"* ตามโครงสร้างเดิมของระบบ

### 16.3 หน้ารวมอีเวนท์ที่บันทึกไว้ (Events Bookmarks — `Bookmarks-Event-1.jpg` & `Bookmarks-Event-2.png`)
* **Structural Layout:** แบ่งพาร์ทการแสดงผลออกเป็น 2 ส่วนหลักในหน้าเดียว (สามารถกดยุบ/ขยายได้ด้วยไอคอนลูกศร `⌃` / `⌵` ตรงกลางหน้าจอ)
* **Upcoming Events:** แสดงการ์ดขนาดใหญ่ (สไตล์เดียวกับหน้าแรกของแอป) ของอีเวนท์ที่กำลังจะเกิดขึ้นเร็ว ๆ นี้ มีฟังก์ชันเปิด Dropdown จัดเรียงตาม Name, Marked Date, หรือ Event Date
* **All Events:** แสดงผลในรูปแบบปฏิทิน (Calendar View) รายเดือนเพื่อให้ผู้ใช้เลือกจิ้มดูตามวัน และด้านล่างจะแสดงเป็น List รายการอีเวนท์แบบเรียงลำดับเวลา (Timeline/Date List Format)
* **Deep Navigation:** เมื่อผู้ใช้กดเลือกที่การ์ดอีเวนท์หรือแถวรายการอีเวนท์ใด ๆ ระบบจะนำทางเชื่อมต่อไปยัง *"หน้าข้อมูลอีเวนท์นั้น ๆ (Event Detail View)"* ตามปกติ

### 16.4 หน้ารวมของแจกที่บันทึกไว้ และตรรกะพินโพสต์ตัวเอง (Giveaways Bookmarks — `Bookmarks-Giveaway-1.png` & `Bookmarks-Giveaway-2.jpg`)
* **Structural Layout:** เลย์เอาต์ถูกแบ่งครึ่งบน-ครึ่งล่างคล้ายกันกับหน้ารวมอีเวนท์:
    * **Upcoming Giveaways:** การ์ดแนวตั้ง 2 คอลัมน์ ด้านบนมีแถบฟิลเตอร์คัดกรองตาม Status, Event, Artist และปุ่มสลับแท็บประเภทโพสต์ (All, Exchange, Non-Exchange)
    * **All Giveaways:** แสดงเป็นรายการ List แถวยาวแนวตั้งลงไปด้านล่าง
* **Ownership Pinning Logic (ตรรกะระบุโพสต์ของฉัน):**
    * การ์ดของแจกใบไหนที่เป็นโพสต์ที่ตัวผู้ใช้เป็นคนสร้างขึ้นมาเอง บนการ์ดใบนั้นจะมี **"ไอคอนหมุดปัก (Pin Icon 📌)"** แสดงที่มุมขวาบนของการ์ด และแสดงชื่อแอปเคานต์เจ้าของเป็น **My Account** (`Bookmarks-Giveaway-1.png`) เพื่อให้แยกแยะออกจากโพสต์บุ๊กมาร์กของคนอื่นได้อย่างชัดเจน
* **Deep Navigation:** เมื่อกดเลือกที่โพสต์ของแจกใด ๆ (ไม่ว่าจะโพสต์ของคนอื่นหรือโพสต์ที่เรา Pin ไว้เอง) ระบบจะนำทางตรงเข้าสู่ *"หน้ารายละเอียดโพสต์ของแจกนั้น ๆ (Giveaway Post Detail View)"* เสมอ
* **Event-Driven Sorting:** ตัวระบบภายในจะไม่ใช้ระบบสุ่มหรือเรียงตามวันที่สร้างโพสต์ในส่วนของ Upcoming Giveaways
* **The Validation Rule:** ระบบจะประมวลผลเช็กค่า Event Date (วันที่จัดงานคอนเสิร์ต/แฟนมีตติ้ง) ของอีเวนท์ที่โพสต์นั้น ๆ ไปผูกสิทธิ์อยู่ โดยจะทำการคัดเลือกและดึงโพสต์ Giveaway เหล่านั้นขึ้นมาแสดงผลในแถบด้านบน ก็ต่อเมื่ออีเวนท์นั้น **"กำลังจะจัดขึ้นเร็ว ๆ นี้ หรือกำลังจัดงานอยู่ในวันนี้"** เท่านั้น เพื่อให้ผู้ใช้สามารถโฟกัสกับของแจกที่ตัวเองต้องไปตามเก็บในวันงานได้ทันท่วงที ซึ่งเป็นตรรกะแบบเดียวกันกับพาร์ทของ Upcoming Events

## 🔔 MODULE 17: NOTIFICATION CENTER & AGGREGATED DETAIL VIEWS

ระบบแจ้งเตือนที่ช่วยจัดหมวดหมู่เหตุการณ์สำคัญและยุบรวมกลุ่มกิจกรรมประเภทเดียวกันเพื่อไม่ให้หน้าจอแจ้งเตือนล้นเกินไป ควบคู่กับระบบควบคุมสต็อกสินค้าเมื่อมีการยืนยันแลกเปลี่ยนของแจก

### 17.1 ประเภทของการแจ้งเตือนหลัก (Notification Types Overview — `Notification-1.png`)
หน้าหลักของระบบแจ้งเตือนจะแสดงผลรายการในรูปแบบ List Card แยกรวมตามหมวดหมู่และประเภทเหตุการณ์หลัก ๆ 7 ประเภท ดังนี้:
1. **Giveaway Ready Notify (กระดิ่ง 🔔):** แจ้งเตือนไปยังแฟน ๆ ที่ทำการ Marked โพสต์ของแจกนั้น ๆ ไว้ ว่าขณะนี้เจ้าของโพสต์ได้เริ่มทำการแจกของแล้ว (ระบบเปลี่ยนสถานะเป็น Ready)
2. **New Artist Success (กระดิ่ง 🔔):** การแจ้งเตือนจากระบบส่วนกลาง (System Notice) แจ้งว่าโปรไฟล์ศิลปินใหม่ที่เราเคยส่งคำขออนุมัติสร้างไป บัดนี้ระบบได้ตรวจสอบและสร้างสำเร็จเสร็จสิ้นแล้ว
3. **Exchange Requested (ลูกศรไขว้ ⇄):** แจ้งเตือนฝั่งเจ้าของโพสต์ว่า มีผู้ใช้คนอื่นส่งคำขอรีเควสขอแลกเปลี่ยนของแจกด้วย
4. **Exchange Accepted (ใบโคลเวอร์ 🍀):** แจ้งเตือนฝั่งผู้ส่งคำขอแลกเปลี่ยน ว่าคำขอที่ส่งไปได้รับการตอบรับ (Accept) จากเจ้าของโพสต์ปลายทางแล้ว
5. **Faved Your Post (ดาว ⭐):** แจ้งเตือนเมื่อมีคนมากด Favorite หรือถูกใจโพสต์ของแจกของเรา
6. **Marked Your Post (ริบบิ้น 🔖):** แจ้งเตือนเมื่อมีคนมากดเซฟ/บุ๊กมาร์ก (Marked) เพื่อติดตามสถานะโพสต์ของแจกของเรา
7. **Replied Your Post (ลูกศรเลี้ยว ↩):** แจ้งเตือนเมื่อมีผู้ใช้เข้ามาพิมพ์ข้อความตอบกลับธรรมดาใต้โพสต์ของเรา

### 17.2 ตรรกะการยุบรวมกลุ่มและการดูหน้ารายละเอียดแยก (Notification Aggregation & Drill-Down)
* **The Aggregation Rule (ตรรกะการรวมกลุ่ม):** เพื่อป้องกันการแจ้งเตือนล้นหลาม หากมีผู้ใช้หลายคนเข้ามากดกระทำแบบเดียวกัน (เช่น กด Fav, กด Marked หรือส่ง Exchange Request เข้ามาพร้อม ๆ กันในระยะเวลาใกล้เคียงกัน) ระบบจะทำการรวมกลุ่ม (Group/Aggregate) กิจกรรมเหล่านั้นให้อยู่ภายใต้การ์ดแจ้งเตือนใบเดียวในหน้าแรกก่อน โดยบนหัวการ์ดจะแสดงรูปอวตารโปรไฟล์ของผู้ใช้เรียงซ้อนกัน (**Stacked Avatars**) แทนการระบุชื่อรายบุคคล
* **Drill-Down Navigation (การกดเข้าดูรายละเอียดแยกแต่ละกลุ่ม):** เมื่อผู้ใช้กดที่การ์ดแจ้งเตือนที่ถูกรวมกลุ่ม ระบบจะนำทางเข้าสู่หน้าย่อยเพื่อคลี่รายการดูรายชื่อทั้งหมดแยกต่างหากอย่างละเอียดแบ่งตาม 2 เลย์เอาต์ดังนี้:

#### A. หน้าแยกกลุ่มประเภท Interaction/Post (`Notification-2.png`)
* *ตัวอย่าง:* หน้าเจาะลึกกลุ่ม Faved your post หรือ Marked your post
* *Layout:* ส่วนหัวจะดึงตัวการ์ดแจ้งเตือนหลักมาปักไว้ด้านบนสุด ถัดลงมาด้านล่างจะสลายกลุ่มออกมาเป็นรายการแถวแนวตั้ง (Vertical Row List) แสดงรายชื่อบัญชีผู้ใช้แต่ละคน (Account `@account`) พร้อมรูปอวตารโปรไฟล์ที่เข้ามามีส่วนร่วมกับโพสต์นั้นแบบเรียงตัวเดี่ยว ๆ

#### B. หน้าแยกกลุ่มประเภทธุรกรรมแลกเปลี่ยน (`Notification-5.png`)
* *ตัวอย่าง:* หน้าเจาะลึกกลุ่ม Exchange Accepted หรือ Exchange Requested
* *Layout:* ส่วนหัวจะดึงตัวการ์ดหลักปักไว้ด้านบนสุดเช่นกัน แต่แถวรายการด้านล่างจะแสดงผลในรูปแบบการ์ดธุรกรรม (Transaction Card) รายคนอย่างชัดเจน
* *Interactive Components:* ภายในแต่ละแถวการ์ดจะประกอบด้วย:
    * ข้อมูลของผู้แลกเปลี่ยน (Account `@account` และระยะเวลา)
    * การ์ดแสดงสิ่งของที่จะนำมาแลกเปลี่ยน (Giveaway และชื่ออีเวนท์)
    * ปุ่ม Action คู่ด้านล่าง:
        * ปุ่ม `[Go to Bookmark]` เพื่อเปิดดูข้อมูลของแจกชิ้นนั้นในคลัง
        * ปุ่ม `[Chat]` เพื่อเปิดหน้าต่างสนทนาแชทส่วนตัวกับผู้ใช้คนดังกล่าวทันทีเพื่อนัดพิกัดรับของ

### 17.3 Sub-Module: Exchange Requested Process

#### 17.3.1 หน้ารวมคำขอแลกเปลี่ยน (Exchange Requested Drill-Down — `Notification-3.png`)
* **UI Layout:** เมื่อกดเข้ามาดูแจ้งเตือนประเภท exchange requested จะแสดงรายการการ์ดคำขอของผู้ใช้คนอื่น ๆ ที่ส่งข้อความและแนบสิ่งของมาขอแลกกับเรา
* **Action Buttons:** ในแต่ละการ์ดคำขอจะมีปุ่มหลัก 2 ปุ่มด้านล่าง:
    * **ปุ่ม `[Reject]` (ปฏิเสธ):** เมื่อกดแล้ว ระบบจะทำการปฏิเสธคำขอแลกเปลี่ยนนั้นทันที สถานะของปุ่มกดบนการ์ดใบนั้นจะถูกเปลี่ยนเป็น Disabled (ไม่สามารถกดซ้ำได้อีก) เพื่อแสดงให้รู้ว่าคำขอนี้ถูกจัดการไปแล้ว
    * **ปุ่ม `[Exchange]` (ตกลงแลกเปลี่ยน):** เมื่อกดแล้ว ระบบจะเปิดหน้าต่าง Modal คอนเฟิร์มการแลกเปลี่ยนและเริ่มคำนวณตัดสต็อกสินค้าทันที

#### 17.3.2 ตรรกะการยืนยันและการตัดยอดสต็อก (Confirmation Modal & Inventory Logic — `Notification-4.png`)
เมื่อผู้ใช้กดปุ่ม `[Exchange]` ระบบจะแสดงผล Confirmation Modal ซ้อนขึ้นมาเหนือหน้าจอหลัก โดยมีรายละเอียดการทำงานเชิงลึกดังนี้:

* **Visual Interface:**
    * แสดงของแจกของฝั่งผู้ขอแลก (Giveaway) อยู่ด้านบน เชื่อมต่อด้วยไอคอนลูกศรสลับทิศทาง `⇅`
    * แสดงของแจกของเรา (My Giveaway) อยู่ด้านล่าง พร้อมตัวเลขจำนวนคงเหลือ (เช่น 20 ea) และตัวระบุการหักลบจำนวนทันที `-1` เด่นชัด

* **Automatic Stock Deduction (ตรรกะการตัดกั๊กยอด):**
    * **Immediate Action:** ทันทีที่ระบบเปิด Modal นี้ขึ้นมาและบันทึกสถานะ "Exchange Booked" ระบบฐานข้อมูล (Database) จะต้องวิ่งไปทำการ หักยอดจำนวนของแจก (Stock Quantity) ในโพสต์ต้นทางของผู้ใช้ลดลงไป 1 ชิ้นทันที ผ่านสมการ:
      $$Stock = Stock - 1$$
    * **Concurrency Control:** เพื่อป้องกันระบบพังในกรณีที่มีคนกดแลกเข้ามาพร้อมกันในเสี้ยววินาที ระบบหลังบ้านต้องตรวจสอบเงื่อนไขก่อนหักว่า หากยอดสต็อกปัจจุบันเท่ากับ 0 ระบบต้องไม่อนุญาตให้เปิด Modal นี้ และต้องแจ้งเตือนว่าของแจกหมดแล้ว

* **Post-Confirmation Navigation (โฟลวหลังจากขึ้นข้อความ "Exchange booked! Let's chat to make an appointment."):**
    * **ปุ่ม `[Close]`:** ทำหน้าที่ปิดหน้าต่าง Modal นี้ลงทันทีเพื่อกลับสู่หน้าเดิม โดยไม่มีการเปลี่ยนหน้าไปที่อื่น (**No Navigation**) เพื่อรักษาบริบท (Context) ให้ผู้ใช้ยังคงอยู่บนหน้าจอรายการแจ้งเตือนคำขอแลกเปลี่ยน (Exchange Requested Sub-screen) และสามารถกดเคลียร์หรือบริหารจัดการคิวคำขอแลกเปลี่ยนของคนอื่น ๆ ในลิสต์รายการถัดไปต่อได้ทันที
    * **ปุ่ม `[Chat]`:** ระบบจะนำทางผู้ใช้ตรงเข้าสู่ห้องแชทส่วนตัว (Direct Message) เพื่อเปิดการสนทนาและนัดหมายรายละเอียดพิกัดเวลาในการรับ-ส่งของแจกกับผู้ขอแลกคนนั้น ๆ ทันที

## 📌 MODULE 18: USER PROFILE DASHBOARD & ACTIVITY TRACKING SPECS

### 18.1 โครงสร้างข้อมูลพื้นฐานและการแก้ไขโปรไฟล์ (Profile Identity & Edit Mode ด้วย Bootstrap Layout)
* **Header & Avatar Configuration (`Profile-2.png`, `Profile-1.png`):**
    * **Header Image:** พื้นหลังแบบรองรับขนาดสากล ใช้คลาส Bootstrap CSS `.w-100` ผู้ใช้สามารถเลือกคลิกอัปโหลดเพื่อปรับเปลี่ยนภาพภูมิหลังได้ผ่านองค์ประกอบคอนเทนเนอร์
    * **Avatar Image:** รูปทรงกลมตัดสไตล์สวยงามด้วยคลาส `.rounded-circle` จัดตำแหน่งทับซ้อนกึ่งกลางแผง เพื่อทำหน้าที่เป็นฐานรองรับตรา Badge สัญลักษณ์ใบโคลเวอร์ในการแชทได้อย่างลงตัว
* **Profile Edit Action:** มีปุ่มกด `[Edit Profile]` (ใช้คลาสสีพื้นฐานของ Bootstrap `.btn .btn-outline-primary .btn-sm`) เพื่อสลับ Component หน้าจอเข้าสู่กล่องแบบฟอร์มแก้ไขข้อมูลพื้นฐานทั่วไป เช่น ชื่อที่ต้องการให้แสดง (Display Name), ข้อความบรรยายแนะนำตัว (Bio) และลิงก์ช่องทางการติดต่อ

### 18.2 ระบบแบ่งเซกชันหลัก (Main Tab Navigation — `Profile-Edit.png`)
ใช้ระบบแถบเมนูนำทางยอดนิยมของ Bootstrap ประกอบเข้ากับ React UI State เพื่อการตอบสนองที่ไหลลื่น:
* **แท็บ Feed (ฟีดโพสต์ส่วนตัว):** วนลูปแสดงผลรายการฟีดการ์ดโพสต์ทั้งหมดที่บัญชีนี้เป็นเจ้าของไอเทม โดยเขียนคำสั่งฝั่งไคลเอนต์จัดเรียงลำดับเวลาจากเวลาล่าสุดย้อนไปเวลาอดีตเสมอ (**Sorted by Latest Timestamp**)
* **แท็บ Activity (บันทึกกิจกรรมย้อนหลัง):** แผงศูนย์รวมข้อมูลดิบกิจกรรมการใช้งานของผู้ใช้ โดยใช้สถาปัตยกรรม Grid ของ Bootstrap จัดสรรและแบ่งกลุ่มย่อยออกเป็น 4 ฟังก์ชันตามภาพอ้างอิงอาร์ตเวิร์ก (`Profile-3.png` ถึง `Profile-6.png`)

### 18.3 เจาะลึกโครงสร้างย่อยในแท็บ Activity (Activity Sub-Sections Logic ใน React)
เมื่อผู้ใช้คลิกสลับค่า State มาที่แท็บ Activity ระบบจะเรนเดอร์คอมโพเนนต์ย่อยตามหมวดหมู่ประวัติย้อนหลังดังนี้:
* **18.3.1 Favorites Log (รายการที่ติดดาวไว้ — `Profile-3.png`):** ดึงประวัติข้อมูลรายการโพสต์กิจกรรมที่ผู้ใช้เคยคลิกเลือกเก็บไอคอน Fav/ดาว ถูกใจเอาไว้ เพื่อความสะดวกในการติดตามดูข้อมูลอัปเดตแบบรวดเร็ว
* **18.3.2 Check-In History (ประวัติการเช็กอินอีเวนต์ — `Profile-4.png`):** ลิสต์รายการการ์ดแสดงรายชื่อคอนเสิร์ตหรืออีเวนต์ศิลปินทั้งหมดที่ผู้ใช้เคยเดินทางไปถึงพิกัดสถานที่จริงและผ่านการกดปุ่มบันทึกยืนยันสิทธิ์สำเร็จ
* **18.3.3 My Giveaways (ประวัติของที่ตัวเองแจก — `Profile-5.png`):** รายการโพสต์คลังของแจกทั้งหมดที่บัญชีนี้เปิดตัวเป็นผู้ใจดี (Giver) พร้อมเครื่องมือ Bootstrap สรุปผลสรุปจำนวนยอดสต็อกที่ทำการสแกนจ่ายออกไปแล้วหน้างานคอนเสิร์ต
* **18.3.4 Received Giveaways (ประวัติของแจกที่เคยไปรับมา — `Profile-Edit.png`, `Profile-6.png`):** รายการคลังรวบรวมของแจกทั้งหมดที่บัญชีนี้เคยยื่นสแกนคิวอาร์รับของจากผู้แจกท่านอื่น ทั้งรูปแบบรับฟรีหน้างาน (General Claim) และรูปแบบข้อตกลงแลกเปลี่ยนสิทธิ์ล่วงหน้า (Exchange Mode) เพื่อเก็บรวบรวมสะสมผลงานแต่ละอีเวนต์

### 18.4 สิทธิ์การเข้าถึงและการแสดงผลโปรไฟล์ของผู้อื่น (Visitor View Logic)
* **ช่องทางการเข้าถึง (Navigation Entries):** ลิงก์นำทางจัดการผ่านตัวควบคุมของ React Router ผู้ใช้สามารถกดเข้าชมหน้าโปรไฟล์บัญชีอื่น (Public Profile) ได้จาก 2 ช่องทางหลักคือ 1) จากหน้าโพสต์แจกของ (คลิกที่ภาพ Avatar ของผู้โพสต์) และ 2) จากหน้าต่างห้องแชท (คลิกที่ภาพโปรไฟล์ขอบบนห้องแชทหรือรูปโปรไฟล์ประกอบข้อความ)
* **กฎการจำกัดสิทธิ์และการซ่อนข้อมูลความเป็นส่วนตัว (Privacy & Permission Constraints ใน React Component):**
    * **No Edit Action:** ปุ่มกด `[Edit Profile]` จะถูกบล็อกด้วยเงื่อนไขทางตรรกะ ไม่ให้มีการเรนเดอร์ลงอินเตอร์เฟสอย่างเด็ดขาด ผู้เยี่ยมชมทั่วไปจะไม่มีสิทธิ์แก้ไขข้อมูลสัญญลักษณ์ใด ๆ ของบุคคลอื่นได้
    * **Favorites Log Hiding:** ในแถบแท็บประวัติกิจกรรม (Activity) เงื่อนไขฟังก์ชันใน React จะทำการตรวจสอบสิทธิ์และ **ตัดการแสดงผลหมวดหมู่ Fav (Favorites) ออกไปจาก DOM โดยสิ้นเชิง** เพื่อปกป้องความเป็นส่วนตัวสูงสุดของผู้ใช้
    * **Visible Lists (ข้อมูลที่เปิดเผยได้):** ระบบจะยินยอมให้เรนเดอร์ข้อมูลกิจกรรมสาธารณะของผู้อื่นได้เพียง 3 หมวดหมู่เท่านั้น ได้แก่ Events Participate (ประวัติเช็กอินอีเวนต์), Giveaways Backup (ของที่เปิดแจก) และ Giveaways Collection (ของที่เคยไปรับมา) ซึ่งถูกจัดรูปเล่มด้วยโครงสร้างสไตล์ Bootstrap อย่างเป็นระบบ

## 📌 MODULE 19: SYSTEM SETTINGS & ACCOUNT MANAGEMENT SPECS

### 19.1 หน้าต่างเมนูหลักการตั้งค่า (Main Settings Menu ด้วย Bootstrap List Group — `Setting-1.png`)
เมื่อกดไอคอนฟันเฟืองจากหน้าโปรไฟล์ ระบบ React Router จะนำทางผู้ใช้สลับมาสู่คอมโพเนนต์หน้า Main Settings ซึ่งใช้สถาปัตยกรรมข้อมูลจัดเรียงโครงสร้างแนวตั้งแบ่งเป็น 3 เซกชันหลัก:
1. **Your Account:** กลุ่มเมนูจัดการตั้งค่าความปลอดภัย ข้อมูลประจำตัวแพลตฟอร์ม และปุ่มคำสั่งออกจากระบบคลังข้อมูล
2. **Notification:** กลุ่มตัวเลือกสำหรับควบคุมเปิด-ปิดการแจ้งเตือนและการสตรีมสัญญาณข้อมูลของแอปพลิเคชัน
3. **Help & Feedback:** ศูนย์บริการสนับสนุนข้อมูล ตอบคำถามสากล และช่องทางรับเรื่องรายงานปัญหาบั๊กจากหน้างาน

### 19.2 โครงสร้างภายในเมนูบัญชีผู้ใช้ (Your Account Sub-Menu — `Setting-2.png`)
เมื่อคลิกเข้ามาที่เมนู Your Account ระบบจะวาดหน้าต่างรายการด้วยคลาส Bootstrap Layout แสดงสิทธิ์การจัดการความปลอดภัย 4 ส่วน:
* **Connected Account (การเชื่อมต่อบัญชี):** แถวแสดงผลแพลตฟอร์มโซเชียลหลักที่ผูกบัญชีใช้งาน (X OAuth 2.0 พร้อมเครื่องหมายถูกและสไตล์ข้อความแสดงสถานะสำเร็จ `✓ Connected`) เพื่อเป็นเครื่องยืนยันความน่าเชื่อถือของแฟนคลับ
* **Country (ภูมิภาค/ประเทศ):** แถวระบุตำแหน่งภูมิภาคปัจจุบันของผู้ใช้ (เช่น Thailand) สามารถกดคลิกเพื่อเปิดกล่องตัวเลือกปรับเปลี่ยนพิกัดการรับชมฟีดเนื้อหาคอนเสิร์ตศิลปินได้
* **Deactivate Account (การปิดฟังก์ชันบัญชีชั่วคราว):** เมนูสิทธิ์พิเศษสำหรับแฟนคลับที่ต้องการระงับการใช้งานโปรไฟล์หรือล้างข้อมูลประวัติส่วนบุคคลออกจากระบบฐานข้อมูลเพื่อความปลอดภัย
* **Log Out Button:** ปุ่มคำสั่งออกจากระบบ (Log Out) ดีไซน์ด้วยสไตล์ Bootstrap แคปซูลสีฟ้าหม่น (`.btn .btn-secondary .rounded-pill`) จัดวางตำแหน่งสมดุลตรงกึ่งกลางด้านล่าง เมื่อคลิกจะทำลายตัวแปรหน่วยความจำ Token และเคลียร์สถานะล็อกอินทันทีเพื่อความปลอดภัย

### 19.3 โครงสร้างภายในเมนูแจ้งเตือน (Notification Sub-Menu — `Setting-3.png`)
เมื่อผู้ใช้กดเข้ามาที่หัวข้อ Notification หน้าจอจะแสดงรายการควบคุมแบบสวิตช์เปิด-ปิด (ใช้คลาสสวิตช์สวยงามของ Bootstrap `.form-check .form-switch`) แบ่งเลเยอร์ระดับการควบคุมออกเป็น 2 ชั้นหลัก:
* **19.3.1 Master Switch (Allow notification):** สวิตช์หลักสำหรับอนุญาตหรือระงับการแจ้งเตือนในภาพรวมระดับแอปพลิเคชัน พ่วงตัวอักษรคำแนะนำด้านล่างระบุคำว่า *“go to your application setting”*
    * *Backend Logic:* หากสลับตั้งค่าสถานะสวิตช์นี้เป็นปิด (Toggle Off) ระบบแอปพลิเคชัน React จะระงับและปิดกั้นการรับสัญญาณส่งผ่าน Push Notification ทุกประเภทที่วิ่งมาจากเซิร์ฟเวอร์หลังบ้านเข้าสู่เครื่องทันที
* **19.3.2 Sub-Category Toggles (Push notification):** ในกรณีที่เปิดสวิตช์หลักทิ้งไว้ ผู้ใช้สามารถเลือกคลิกเปิดปิดเพื่อกรองประเภทหัวข้อการรับรู้แยกอิสระได้ 3 หมวดหมู่ย่อย:
    1. *Related to your post:* ข้อความแจ้งเตือนที่มีความเกี่ยวข้องโดยตรงกับโพสต์ไอเทมแจกของตนเอง (เช่น มีแฟนคลับกดสแกนรับของสำเร็จ, มีผู้ใช้มาพิมพ์แสดงความเห็นต่อท้ายเธรด)
    2. *Other post related:* ข้อความเกี่ยวกับโพสต์แจกของของสมาชิกท่านอื่นที่ตนเองมีส่วนร่วม (เช่น โพสต์ศิลปินที่เคยกดติดดาว Fav บันทึกไว้ หรือการแจ้งอัปเดตสถานะของโพสต์แลกเปลี่ยนของกัน)
    3. *Chat:* การยิงเตือนเมื่อมีข้อความแชทใหม่ส่งเข้ามาในกล่อง Inbox (โดยห้องแชทไหนที่มีการกดปุ่มกระดิ่งตั้งค่าปิดเสียง Mute รายห้องเอาไว้ ระบบจะทำการซ้อนตัวกรองไม่ส่งสัญญาณเตือนซ้ำซ้อนตามกฎเกณฑ์สากล)

### 19.4 โครงสร้างภายในเมนูช่วยเหลือและคำติชม (Help & Feedback Sub-Menu — `Setting-4.png`)
หน้าต่างศูนย์รวมการให้บริการสนับสนุนข้อมูลและแก้ไขความขัดแย้งหน้างาน ประกอบด้วย 3 ส่วนโครงสร้าง:
* **19.4.1 Application Version:** แถวแสดงรหัสเวอร์ชันปัจจุบันของแอปพลิเคชันบนเครื่อง (ระบบเริ่มต้นระบุตัวอักษร: `v 1.00`) เพื่อใช้เป็นข้อมูลอ้างอิงส่งประสานงานทีมเทคนิคเมื่อเกิดข้อผิดพลาด
* **19.4.2 Help (ศูนย์ช่วยเหลือเชิงคู่มือ):** เมนูทางเลือกแบบกดคลิกเพื่อเชื่อมต่อไปยังหน้าเนื้อหาข้อมูล (เรนเดอร์แถวจัดรูปเล่มด้วย Bootstrap `.list-group-item-action`) ประกอบด้วยคู่มือย่อย:
    * *How to add to home screen:* แนะนำวิธีนำระบบเว็บแอปพลิเคชัน (PWA) ติดตั้งปักหมุดไว้บนหน้าจอหลักของระบบปฏิบัติการสมาร์ทโฟน
    * *Community Rules:* หน้ารายละเอียดกติกา ข้อตกลงร่วมกันของคอมมูนิตี้ เพื่อรักษามาตรฐานความปลอดภัยและความโปร่งใสในการแจกของรางวัลร่วมกัน
* **19.4.3 Feedback (ช่องทางแจ้งปัญหา):** รายการลิงก์เชื่อมต่อระบบส่งข้อมูลกลับหาทีมพัฒนาแอปพลิเคชัน:
    * *Report a problem:* แบบฟอร์มรายงานพฤติกรรมผิดปกติ บั๊กในระบบคอมโพเนนต์ หรือช่องทางด่วนสำหรับแจ้งเหตุทุจริต/โกงกันหน้างานคอนเสิร์ต
    * *Suggestions:* ช่องทางเปิดรับแนวคิดสร้างสรรค์ คำแนะนำไอเดียดี ๆ ที่อยากให้ทีมงานพัฒนาเพิ่มเติมในระบบเวอร์ชันถัดไป

## 📌 MODULE 20: DEEP LINKING & GUEST MODE RESTRICTIONS (AUTHENTICATION REDIRECT)

### 20.1 สิทธิ์การรับชมโพสต์ผ่านลิงก์แชร์ภายนอก (Guest Mode Viewer ใน React Router)
* **Public Post Preview:** เมื่อบุคคลภายนอกกดคลิกลิงก์แชร์ที่ส่งต่อกันมา (เช่น เส้นทาง URL รูปแบบ `/giveaway/:id`) โครงสร้างระบบ Routing ใน React Router จะยินยอมให้ประมวลผลเรนเดอร์และเปิดการแสดงผลเนื้อหาโพสต์แจกของนั้นได้ตามปกติภายใต้สถานะผู้เยี่ยมชมสาธารณะ "Guest Mode" แม้ว่าบัญชีนั้นจะยังไม่ได้ผ่านขั้นตอนลงชื่อเข้าใช้งานระบบ (Logged Out / Unauthenticated) ก็ตาม
* **Action Interception (การบล็อกกิจกรรม):** ยูสเซอร์ที่อยู่ในสถานะ Guest Mode จะมีสิทธิ์ใช้งานแอปพลิเคชันในระดับ **"ดูข้อมูลได้อย่างเดียวเท่านั้น (Read-Only)"** โดยฟังก์ชันใน React Component จะตรวจสอบสถานะล็อกอินและเขียนเงื่อนไขสกัดกั้น ระงับการทำกิจกรรมโต้ตอบเชิงรุกกับโพสต์แจกของโดยสิ้นเชิง:
    * ไม่สามารถกดสัญลักษณ์ดาว/หัวใจเพื่อบันทึกโพสต์ (Fav)
    * ไม่สามารถกดคลิกส่งปุ่มจองสิทธิ์แลกเปลี่ยนล่วงหน้า (Marked / Reserved)
    * ไม่สามารถกดเรียกสิทธิ์เพื่อขอเปิดคิวอาร์โค้ดรับของหน้างาน หรือกดปุ่มเข้าร่วมกิจกรรมใด ๆ ได้ทั้งสิ้น

### 20.2 ระบบนำทางเพื่อบังคับลงชื่อเข้าใช้ (Authentication Gate & Redirect Logic)
* **Trigger Link to Get Started:** ทันทีที่ผู้ใช้ในสถานะ Guest Mode พยายามฝืนกดคลิกปุ่มกระทำกิจกรรมโต้ตอบ (Action Buttons) ใด ๆ บนหน้าต่างโพสต์ (เช่น พยายามกดดาว Fav หรือกดปุ่มส่งคำขอจองสิทธิ์) ฟังก์ชันใน React Handler จะทำการสกัดกั้นขัดจังหวะอีเวนต์ทันที และสั่งเปลี่ยนเส้นทาง Navigation พายูสเซอร์พุ่งตรงไปยังหน้าจอเริ่มต้นใช้งาน **"Get Started / Welcome Screen" (Module 1)** ทันที เพื่อบังคับให้ผ่านกระบวนการ Log In หรือ Sign Up ผ่านบัญชี X ให้เสร็จสิ้นเรียบร้อยก่อน
* **Intelligent Return Navigation (ระบบจำลิงก์เดิมด้วย React Router State):**
    * **กรณีที่ทำได้ (Ideal Flow):** ในจังหวะที่คำสั่งขัดจังหวะทำการเด้งตัวผู้ใช้ไปยังหน้าเริ่มต้นใช้งาน ให้โปรแกรมทำการแนบและฝังพารามิเตอร์ประวัติตำแหน่งลิงก์โพสต์เดิมพ่วงเข้าไปกับสถานะการนำทาง (ฝังค่าไว้ในพารามิเตอร์ `state: { from: location.pathname }` ของ React Router) เมื่อผู้ใช้ทำการผ่านกระบวนการล็อกอินหรือสมัครสมาชิกผ่านระบบ X OAuth 2.0 เสร็จสิ้นสมบูรณ์ดีแล้ว ตัวควบคุมทิศทางจะตรวจเช็กค่าตำแหน่งดังกล่าวและทำการดีดนำพายูสเซอร์คนนั้นเด้งวาร์ปกลับมายังหน้าลิงก์โพสต์แจกของเดิมที่เขาเปิดค้างไว้ตอนแรกสุดทันที เพื่อให้แฟนคลับรายนั้นสามารถดำเนินกิจกรรมกดส่งสิทธิ์ต่อได้ทันทีโดยไม่ต้องไปเสียเวลาพิมพ์ค้นหาโพสต์ใหม่อีกรอบ
    * **กรณีที่เกิดข้อจำกัด (Fallback Flow):** หากเกิดข้อจำกัดทางเทคนิคที่ไม่คาดคิดเกี่ยวกับระบบปฏิบัติการอุปกรณ์ หรือโครงสร้าง Routing เกิดสูญเสียสถานะ State กลางคันในระหว่างสลับไปแอปพลิเคชันภายนอก ทำให้ไม่สามารถดึงค่าพารามิเตอร์ลิงก์เดิมกลับมาใช้งานได้ ทันทีที่กระบวนการเข้าสู่ระบบเสร็จสิ้น ให้โปรแกรมทำการส่งผู้ใช้กลับไปยัง **หน้าจอหลัก (Home/Feed Screen)** ตามโฟลวขั้นตอนปกติทันที เพื่อรักษาเสถียรภาพการทำงานและไม่ทำให้ตัวแอปพลิเคชันเกิดอาการค้างหน้าจอสีขาวเด็ดขาด
